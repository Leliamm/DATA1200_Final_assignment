Accessibility

What is web accesibility?

Web accessibility is the wide known phenomenon that a website should be accessible to as many individuals as possible (Persson et. al. 2015). A website with good web accessibility is easy to access and use for all people. This includes people with and without disabilities. For instance, normal people using other devices with different screen sizes, or people with visual impairments.

For this assignment we attempted to remove barriers which made our website inaccessible.

How did we test web accessibility?

To test web accessibility, we used an automated test, AChecker, on all our pages. The website evaluated our HTML-code and gave us feedback on how we could improve our accessibility. We aimed for the grade AA or higher, since this indicated that there were no known problems with our code.

Even so, we still made active decisions to improve our accessibility beyond what the test showed. The test also gave us a list of potential problems, which we carefully found and assessed. We fixed the ones we thought were applicable to our case.

Things we have changed

Navigation bar

According to the automated check, the navigation bar is a list of links and must be marked appropriately. Therefore, we encapsuled our navigation bar with the semantic tag <ul> and the corresponding <li> for each item on the list. This ensures an isolated section with information, with excellent structure.

"Skip to main content" button (with link)

According to the automated check we needed a “skip to main content” button. This will elevate our pages because our users can easily skip to the main content. Without this button, the process of navigating the website could be a tedious process for various people, especially for our friends with visual impairments.

Change not needed

Format for h1, h2 and h3

According to the automated check, h1, h2 are potentially used for formatting. This is not the case. We used our header elements as section headers to structure our different themes. We only used formatting in a CSS file that is separated from the HTML files that we used in the accessibility check.

Primary language

According to the automated check, our webpages should only have one primary language. However, we used dummy text because it was allowed in our task description. Therefore, we choose to ignore the feedback.

Contradictory feedback

Alt text for images

According to the automated check, we had two potential problems with our “alt” attribute to all our images. First, it assumed our alt descriptions should have been longer. Then it commented that our texts were too long. We choose to disregard both statements. Which lead to a detailed visual description for the users who cannot see them. Our alt text make it a better experience using a screen reader.

Web accessibility beyond the accesibility test

We used other accessibility tests to improve our visually impaired users experience. For instance, we used strong color contrasts for our texts, images, and background images. We tested our website used an online colorblind test to make sure colorblind people had no trouble seeing the different colors in our website. We also made sure to use the right sematic tags, and lines under all our links, so it would be easier to see the links properly. Lastly, we used fonts which makes it easy to read. As a result, people with visual impairments can still enjoy our page, without compromise.

Some other coding changes we made include correct sematic tags and attributes accordingly. We used the <html>“lang” attribute to define our primary language. Furthermore, for all our images, we used the “alt” attribute. Lastly, we used “back to main content” buttons for our website for easy “scrolling” and tried to make our website easy to navigate.
